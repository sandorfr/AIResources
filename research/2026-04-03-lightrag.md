## What is it?

LightRAG is an open-source Retrieval-Augmented Generation framework from the University of Hong Kong that weaves a knowledge graph into the indexing and retrieval pipeline. Where traditional ("naive") RAG splits documents into flat chunks and retrieves them by vector similarity, LightRAG first extracts entities and relationships with an LLM, builds a graph, then searches that graph at query time. The result is answers that capture how ideas connect across hundreds or thousands of pages instead of returning isolated fragments.

The system works in four stages. First, documents are chunked and an LLM extracts entities (nodes) and relationships (edges). Second, a profiling step generates key-value pairs for each node and edge -- keys for fast lookup, values holding summarized text for generation. Third, a deduplication pass merges identical entities from different document segments to keep the graph compact. Fourth, a dual-level retrieval system handles queries: low-level retrieval targets specific entities ("Who authored this paper?") while high-level retrieval aggregates across relationships to answer broader thematic questions.

What sets LightRAG apart is efficiency. It often resolves a query in a single API call where competitors like Microsoft GraphRAG may need hundreds. It also supports incremental updates -- new documents are merged into the existing graph in seconds without rebuilding the entire index. Published at EMNLP 2025, it consistently outperforms baselines on comprehensiveness, diversity, and accuracy across legal, agricultural, and computer science benchmarks.

## When to use it

- **Large document corpora (500+ pages):** LightRAG shines when you have hundreds or thousands of documents where flat vector search loses context. It maintains a connected "mind map" of all your data.
- **Multi-hop reasoning queries:** When answering a question requires synthesizing information scattered across multiple documents -- comparing theories, tracing causal chains, mapping relationships between entities.
- **Dynamic, frequently updated knowledge bases:** The incremental update algorithm lets you add new documents in seconds without re-indexing. Ideal for finance, news, legal, or any domain where data changes fast.
- **Cost-sensitive deployments:** Significantly cheaper per query than GraphRAG. One API call vs. hundreds for comparable results.
- **Enterprise knowledge management:** Internal document repositories, technical documentation, research libraries -- anywhere you need deep, synthesized answers rather than keyword matches.
- **Healthcare and legal research:** Domains with notoriously complex, interconnected information where accuracy and comprehensiveness are critical.
- **Local/private deployments:** Runs entirely locally via Ollama for organizations with strict privacy requirements or limited cloud API budgets.

---

## When NOT to use it / Limitations

**Skip LightRAG when:**
- You have fewer than ~10 documents. Modern LLM context windows (100k+ tokens) can handle small corpora directly without any retrieval system.
- You need multimodal support out of the box. LightRAG handles text and PDFs only. Images, charts, and tables require the separate RAG-Anything extension.
- Your available LLM is under 32B parameters or has less than 32k context. The entity-relationship extraction is demanding and smaller models produce poor graphs.

**Technical limitations:**
- **Initial indexing is slow.** While incremental updates are fast, the first embedding pass over a large library is the most painful part and can take hours.
- **Embedding model lock-in.** Switching embedding models later means clearing the entire data directory and rebuilding from scratch.
- **Hardware requirements.** Running locally at the recommended 32k+ context size requires significant GPU memory. 6GB VRAM cards will struggle.
- **Irreversible deletions.** Deleting entities or documents cannot be undone, and large-scale deletions are slow because the graph must be partially reconstructed.
- **Observability gaps.** Langfuse tracing currently works only with OpenAI-compatible APIs -- Ollama, Azure, and Bedrock are not yet supported.

**Alternatives:** Microsoft GraphRAG (more established but much more expensive), naive RAG with Pinecone/Supabase (simpler but no multi-hop reasoning), or long-context LLMs for small datasets that fit in a single prompt.

## Getting Started

### Install

```bash
# Recommended: using uv
uv tool install "lightrag-hku"

# Or with pip in a virtual environment
python -m venv .venv && source .venv/bin/activate
pip install "lightrag-hku"
```

### Configure

Set your OpenAI API key (or configure Ollama for local use):

```bash
export OPENAI_API_KEY="sk-...your_key..."
```

### First query (Python)

```python
import os, asyncio
from lightrag import LightRAG, QueryParam
from lightrag.llm.openai import gpt_4o_mini_complete, openai_embed

WORKING_DIR = "./rag_storage"
os.makedirs(WORKING_DIR, exist_ok=True)

async def main():
    rag = LightRAG(
        working_dir=WORKING_DIR,
        embedding_func=openai_embed,
        llm_model_func=gpt_4o_mini_complete
    )
    await rag.initialize_storages()  # Don't skip this!

    await rag.ainsert("Your document text here...")

    response = await rag.aquery(
        "What problem does this solve?",
        param=QueryParam(mode="hybrid")  # combines local + global search
    )
    print(response)
    await rag.finalize_storages()

asyncio.run(main())
```

### Web UI

For a visual interface to upload documents and explore the knowledge graph:

```bash
lightrag-server
# Open http://localhost:9621
```

From the dashboard you can upload PDFs, visualize entity-relationship graphs, and test different retrieval modes interactively.
